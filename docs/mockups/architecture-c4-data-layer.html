Architecture Decision Record

Recommendation: Option C — Hybrid Zustand + SQLite

Zustand remains the runtime state engine powering the reactive UI. SQLite (via better-sqlite3) serves as the durable persistence and analytical query layer, running exclusively in the Electron main process. The renderer never touches the database directly — all persistence flows through typed IPC handlers, maintaining strict process isolation.

This hybrid approach preserves Showtime's existing architecture (Zustand stores, IPC bridge, preload API) while adding durability and multi-day query capability without rewiring the entire data flow.

Why this architecture
  • Zustand is already battle-tested in Showtime — zero migration risk for runtime state
  • SQLite's sync API means zero async overhead for reads in the main process
  • Strict IPC boundary enforced by Electron preload — renderer never imports Node.js
  • Debounced writes (5s or phase transitions) prevent write amplification during live shows
  • Historical queries (drift analytics, streak patterns) need SQL, not in-memory filtering
Strengths
  • Incremental adoption: Zustand persist middleware stays, SQLite adds durability underneath
  • timeline_events table enables day-structure drift tracking with a simple query
  • Migrations via versioned SQL files — no ORM overhead
  • Single-file database in app data dir — no server, no network, zero config
  • WAL mode for safe concurrent reads during write batches
Trade-offs
  • Native dependency (better-sqlite3) requires electron-rebuild for each Electron version
  • Two sources of truth during runtime — sync bugs possible if IPC fails silently
  • Schema migrations need careful versioning as the data model evolves
Rejected Alternatives
Option A: SQLite Only
Requires replacing Zustand as the reactive store. Too invasive for v1. Loses Zustand's middleware ecosystem (persist, devtools).
Option B: localStorage + JSON files
Not queryable for historical analytics. IndexedDB is async-only, adding latency to hot-path reads. JSON files have no schema enforcement.
Option D: DuckDB + JSON
DuckDB's Node.js bindings are immature for Electron. Overkill for v1 analytics. Adds a second native dependency with worse Electron support than better-sqlite3.

Schema

Data Model

Five tables: shows, acts, timeline_events, claude_contexts, and migrations. The timeline_events table is the foundation for the day-structure drift feature.

1 row per day

N rows per show
ColumnTypeDescription

T

timeline_events

KEY TABLE — powers drift analytics
ColumnTypeDescription
id INTEGER PK Auto-increment row ID
show_id TEXT FK FK → shows.id
act_id TEXT FK FK → acts.id (NULL for show-level events)
event_type TEXT act_planned | act_started | act_completed | act_skipped | act_extended | beat_locked | beat_skipped | intermission_start | intermission_end | show_started | show_struck | director_entered
timestamp INTEGER Unix ms when event occurred
planned_start INTEGER Unix ms — when act was expected to start (based on lineup order + durations)
planned_end INTEGER Unix ms — planned_start + duration_minutes * 60000
actual_start INTEGER Unix ms — when act actually started (from act.startedAt)
actual_end INTEGER Unix ms — when act actually ended (from act.completedAt)
drift_seconds INTEGER Computed: (actual_start - planned_start) / 1000. Positive = late, negative = early.
metadata TEXT JSON blob for event-specific data (extension minutes, director choice, etc.)

Example: Day Structure Drift Query

-- Get drift report for today's show
SELECT
  a.name                              AS act_name,
  a.sketch                            AS category,
  te.planned_start,
  te.actual_start,
  te.planned_end,
  te.actual_end,
  te.drift_seconds,
  CASE
    WHEN te.drift_seconds BETWEEN -300 AND 300
      THEN 'on_track'
    WHEN te.drift_seconds > 300
      THEN 'late'
    ELSE 'early'
  END                                AS drift_status
FROM timeline_events te
JOIN acts a ON a.id = te.act_id
JOIN shows s ON s.id = te.show_id
WHERE s.date = '2026-03-21'
  AND te.event_type = 'act_started'
ORDER BY a.order_index;

Example: 7-Day Streak & Pattern Query

-- Past 7 days: verdict distribution + beat averages
SELECT
  s.date,
  s.energy,
  s.verdict,
  s.beats_locked,
  s.beat_threshold,
  COUNT(a.id)                       AS total_acts,
  SUM(a.status = 'completed')        AS completed_acts,
  SUM(a.status = 'skipped')           AS skipped_acts,
  AVG(te.drift_seconds)              AS avg_drift_seconds
FROM shows s
LEFT JOIN acts a ON a.show_id = s.id
LEFT JOIN timeline_events te
  ON te.show_id = s.id
  AND te.act_id = a.id
  AND te.event_type = 'act_started'
WHERE s.date >= date('now', '-7 days')
GROUP BY s.id
ORDER BY s.date DESC;

Migration 001: Initial Schema

-- 001_initial_schema.sql
PRAGMA journal_mode = WAL;
PRAGMA foreign_keys = ON;

CREATE TABLE IF NOT EXISTS _migrations (
  version   INTEGER PRIMARY KEY,
  name      TEXT NOT NULL,
  applied_at TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS shows (
  id                    TEXT PRIMARY KEY,
  date                  TEXT NOT NULL UNIQUE,
  phase                 TEXT NOT NULL DEFAULT 'no_show',
  energy                TEXT,
  beats_locked          INTEGER NOT NULL DEFAULT 0,
  beat_threshold        INTEGER NOT NULL DEFAULT 3,
  verdict               TEXT,
  writers_room_step     TEXT DEFAULT 'energy',
  timer_end_at          INTEGER,
  timer_paused_remaining INTEGER,
  current_act_id        TEXT,
  claude_session_id     TEXT,
  created_at            TEXT NOT NULL DEFAULT (datetime('now')),
  updated_at            TEXT NOT NULL DEFAULT (datetime('now'))
);

CREATE TABLE IF NOT EXISTS acts (
  id                TEXT PRIMARY KEY,
  show_id           TEXT NOT NULL REFERENCES shows(id) ON DELETE CASCADE,
  name              TEXT NOT NULL,
  sketch            TEXT NOT NULL,
  duration_minutes  INTEGER NOT NULL,
  status            TEXT NOT NULL DEFAULT 'upcoming',
  beat_locked       INTEGER NOT NULL DEFAULT 0,
  order_index       INTEGER NOT NULL,
  started_at        INTEGER,
  completed_at      INTEGER
);

CREATE TABLE IF NOT EXISTS timeline_events (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  show_id         TEXT NOT NULL REFERENCES shows(id) ON DELETE CASCADE,
  act_id          TEXT REFERENCES acts(id) ON DELETE CASCADE,
  event_type      TEXT NOT NULL,
  timestamp       INTEGER NOT NULL,
  planned_start   INTEGER,
  planned_end     INTEGER,
  actual_start    INTEGER,
  actual_end      INTEGER,
  drift_seconds   INTEGER,
  metadata        TEXT
);

CREATE TABLE IF NOT EXISTS claude_contexts (
  id              INTEGER PRIMARY KEY AUTOINCREMENT,
  show_id         TEXT NOT NULL REFERENCES shows(id) ON DELETE CASCADE,
  session_id      TEXT,
  energy_level    TEXT,
  plan_dump       TEXT,
  lineup_json     TEXT,
  model           TEXT,
  cost_usd        REAL,
  created_at      TEXT NOT NULL DEFAULT (datetime('now'))
);

-- Indexes
CREATE UNIQUE INDEX idx_shows_date ON shows(date);
CREATE INDEX idx_acts_show_id ON acts(show_id, order_index);
CREATE INDEX idx_timeline_show_act ON timeline_events(show_id, act_id);
CREATE INDEX idx_timeline_type ON timeline_events(event_type, timestamp);
CREATE INDEX idx_claude_show ON claude_contexts(show_id);
